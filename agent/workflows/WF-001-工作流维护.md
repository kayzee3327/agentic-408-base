# 工作流维护

## 目标

让本资料库的可重复维护方法沉淀为可被 AI agent 阅读和执行的工作流，而不是只存在于一次对话里。

## 触发条件

- 用户明确要求以后按某种方式执行。
- 当前任务产生了可复用、会反复出现的处理流程。
- 现有工作流不能覆盖当前输入类型或维护动作。
- 模板、脚本、目录结构发生变化，导致原流程需要同步更新。
- 结构检查或实际执行暴露出流程缺口。
- 需要记录流程、维护说明或可重复执行规则时，优先更新 `agent/workflows/`。

## 执行步骤

1. 先阅读 `AGENTS.md` 和 `agent/workflows/README.md`。
2. 判断是新增工作流、修改已有工作流，还是只更新索引。
3. 输出工作流变更报告，说明触发原因、拟修改文件、影响范围和是否涉及知识点内容。
4. 使用 `agent/templates/maintenance/工作流模板.md` 的字段结构创建或更新工作流。
5. 保持变更增量、具体、可执行，不写成泛泛原则。
6. 更新 `agent/workflows/README.md` 中的索引。
7. 运行结构检查脚本。

## 允许主动更新的情况

- 只修改工作流、模板、README 或校验脚本，不改变正式知识点事实。
- 新工作流是对本次已经执行步骤的归纳。
- 修改能减少以后重复解释或降低误操作风险。

## 需要谨慎处理的情况

- 若工作流会改变来源规则、ID规则、大纲更新规则或知识点写入边界，应先明确说明影响，再修改。
- 若用户明确要求只讨论、不落地，则不得主动写入文件。
- 若工作流和 `AGENTS.md` 冲突，以 `AGENTS.md` 为准，并优先修正工作流。
- 若只是记录可重复执行流程，不要另建 `docs/`；应新增或更新 `agent/workflows/` 下的工作流文件。
